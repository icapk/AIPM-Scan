## 四、面试答辩话术

### 4.1 核心问题："市面上有这么多产品，你为什么还要做？"

**标准回答**：

```
这个问题我认真思考过，在做之前分析了4个主要竞品。

【发现问题】
现有产品有3个共同缺陷：

1. 缺乏垂直深度
   - 所有产品都是"万金油"，覆盖所有职位
   - 对AI PM这种专业岗位，评估非常表面
   - 比如，AI PM需要的"AI技术理解"维度，通用面试产品根本没有

2. 评估不可解释
   - 只给简单分数或模糊建议
   - 用户不知道具体哪里需要改进
   - 没有证据句支撑

3. 技术方案粗糙
   - 直接调用LLM生成题目和评估
   - 没有专业知识库支撑
   - 容易产生幻觉

【我的差异化方案】
所以我决定：
- 只做AI PM垂直领域
- 用RAG知识库确保专业性
- 设计六维能力模型结构化评估
- 抽取证据句让评分可解释

【核心价值】
这不是要打败成熟产品，而是：
1. 验证"垂直+RAG"在招聘场景的价值
2. 深入理解AI PM能力模型
3. 证明我的产品思维和技术应用能力

如果有机会，我希望在贵司把这个想法做成真正的B端产品。
```

### 4.2 追问："你的RAG比直接用LLM好在哪？"

```
RAG解决了LLM的3个问题：

1. 准确性
   - LLM直接生成可能有幻觉
   - RAG检索专业知识库，评估有据可依

2. 稳定性
   - LLM同样的输入可能给出不同评分
   - RAG基于知识点匹配，评分更稳定

3. 可解释性
   - LLM是黑盒，不知道为什么给这个分
   - RAG可以展示：基于哪些知识点给分

我在项目中测试发现，用RAG后：
- 同一回答多次评分偏差从±2分降到±0.5分
- 证据句更具体，用户反馈更清晰
```

### 4.3 追问："你的产品能商业化吗？"

```
坦白说，以当前MVP的完成度，还不能直接商业化。

【我的认知】
- 成熟产品有完整团队、资金、用户积累
- 我的MVP只是核心功能验证
- 用户体验和功能完整度有明显差距

【项目价值】
但这个项目对我的价值是：
1. 深入理解AI PM能力模型
2. 掌握RAG+LLM技术应用
3. 展示从0到1的产品思维
4. 为未来B端产品积累经验

【未来可能】
如果要商业化，我的规划是：
1. Phase 2加入用户系统和B端功能
2. 可能开源，服务AI PM社区
3. 也可以在新公司内部使用

这个项目证明的是能力，不是商业成功。
```

## 八、常见质疑回应

### 8.1 项目定位澄清

> **这是求职作品集，不是商业化产品**

| 误区 | 正确理解 |
|-----|---------|
| "应该做完整的B端功能" | MVP 阶段聚焦核心验证，B端是 Phase 2 |
| "六维模型需要专家验证" | 模型展示的是**设计方法论**，不是权威标准 |
| "应该做用户调研" | 求职作品的用户是**面试官**，不是真实客户 |

### 8.2 技术质疑回应

**质疑1："RAG调优很复杂，非工程师做不了"**

```
回应：我用的是 RAGFlow 封装好的工具，不是从零搭建 RAG。
Embedding、Chunk、检索的复杂度已被封装，我只需配置使用。
这就像用 Notion 和自己写数据库的区别。
```

**质疑2："证据句抽取难度很高"**

```
回应：证据句本质是 Prompt 工程，不是算法开发。
通过迭代 Prompt + 知识库优化，完全可以达到演示可用水平。
MVP 不追求完美，追求可展示。
```

**质疑3："六维能力模型是自嗨"**

```
回应：任何能力模型都是从"自定义"开始的。
我基于 20+ 个 AI PM 岗位 JD 提炼出这六个维度。
面试时我展示的是设计方法论，而非宣称这是行业标准。
后续可与 HR 专家共创迭代，这正是 Phase 2 的计划。
```

### 8.3 商业质疑回应

**质疑："B端SaaS经验没有最大化"**

```
回应：B端经验在面试中口述展示即可，不需要全塞进 MVP。
Phase 1 聚焦技术验证，Phase 2 做 B 端功能。
这是正确的阶段划分，而非"放弃差异化"。
```

**质疑："价值主张太技术化"**

```
回应：求职作品的受众是技术面试官，需要展示技术理解力。
商业洞察通过项目背景和竞品分析来体现。
"RAG可解释评估"本身就是商业价值——让招聘决策有据可依。
```
